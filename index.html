Create a free Google Cloud account at cloud.google.com. Google gives you $300 in credits for 90 days plus an always-free tier that never expires. For Cloud Run specifically, the free tier covers 2 million requests/month, 360,000 GB-seconds of compute, and 180,000 vCPU-seconds — more than enough for a portfolio project. You'll need a credit card to verify identity, but won't be charged unless you manually upgrade. 

# Install the Google Cloud CLI (macOS / Linux)
curl -O https://dl.google.com/dl/cloudsdk/channels/rapid/downloads/google-cloud-cli-linux-x86_64.tar.gz
tar -xf google-cloud-cli-linux-x86_64.tar.gz
./google-cloud-sdk/install.sh

# Authenticate and create your project
gcloud auth login
gcloud projects create my-portfolio-project --name="Portfolio"
gcloud config set project my-portfolio-project
✓ Why this step? The gcloud CLI is the main tool for everything that follows. Authenticating now means you can deploy, push images, and manage billing entirely from the terminal.

YOUR NOTES

Cloud Run runs your app as a Docker container. You need two files in your project root: a Dockerfile and a requirements.txt. The most common error here is running docker build from the wrong folder — your terminal must be inside the same directory as your Dockerfile.

⚠ Common error — "no such file or directory" If you see open Dockerfile: no such file or directory, Docker can't find your Dockerfile. Your terminal is in the wrong folder. Run the directory check below first. 
# 1. Confirm you're in the right directory
pwd
# Should show the folder containing your app files

# 2. Check Dockerfile and requirements.txt are present
ls -la
# You should see: Dockerfile  requirements.txt  app.py

# If you're in the wrong place, navigate to your project folder:
cd /path/to/your/project
# requirements.txt — save this in your project folder
flask==3.0.3
gunicorn==22.0.0
# Dockerfile — no file extension, saved in your project folder
FROM python:3.12-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

ENV PORT=8080
EXPOSE 8080

CMD ["gunicorn", "--bind", "0.0.0.0:8080", "app:app"]
# Verify both files exist before moving to Step 3
ls Dockerfile requirements.txt
# Expected: Dockerfile    requirements.txt
# If you see "No such file" — the file isn't saved in this folder yet
✓ Why this step? python:3.12-slim keeps the image small for faster deploys. Gunicorn is a production-grade WSGI server — never use Flask's built-in dev server in production. The PORT env variable lets Cloud Run dynamically assign the port.

YOUR NOTES

Google Artifact Registry stores your Docker images privately. You build the image locally and push it to a repository in your GCP project. Cloud Run pulls from that registry during deployment — no third-party registry credentials needed. Complete Step 2 fully before running anything here — the push error tag does not exist means the build in Step 2 didn't succeed yet.

⚠ Common error — "tag does not exist" If you see tag does not exist: .../my-app:latest on push, the Docker build in Step 2 failed or was never run. There is no image to push yet. Fix the build first, then come back here. 
# Enable required APIs (one-time)
gcloud services enable artifactregistry.googleapis.com run.googleapis.com

# Create a Docker repository in Artifact Registry (one-time)
gcloud artifacts repositories create portfolio-repo \
  --repository-format=docker \
  --location=us-central1 \
  --description="Portfolio app images"

# Authenticate Docker to push to GCP
gcloud auth configure-docker us-central1-docker.pkg.dev
# Build your image — run this from inside your project folder
# (the dot at the end means "look for Dockerfile here")
docker build -t us-central1-docker.pkg.dev/my-portfolio-project/portfolio-repo/my-app:latest .

# Verify the image was built successfully before pushing
docker images | grep my-app
# Expected: us-central1-docker.pkg.dev/.../my-app   latest   abc123   ...
# If this returns nothing — the build failed. Check the output above for errors.
# Only run this once docker images shows your image above
docker push us-central1-docker.pkg.dev/my-portfolio-project/portfolio-repo/my-app:latest

# Confirm the image is in Artifact Registry
gcloud artifacts docker images list \
  us-central1-docker.pkg.dev/my-portfolio-project/portfolio-repo
# Expected: a row showing my-app with a digest and timestamp
✓ Why this step? Artifact Registry is tightly integrated with Cloud Run — no extra permissions needed. The docker images check before pushing confirms you have a local image to push, catching build failures before they become confusing push errors.

YOUR NOTES

Deploy your image with flags tuned for the free tier. --min-instances 0 means the service scales to zero when not in use — you pay nothing for idle time. --allow-unauthenticated makes it publicly accessible. You get a live HTTPS URL immediately. 

gcloud run deploy my-app \
  --image=us-central1-docker.pkg.dev/my-portfolio-project/portfolio-repo/my-app:latest \
  --platform=managed \
  --region=us-central1 \
  --allow-unauthenticated \
  --min-instances=0 \
  --max-instances=3 \
  --memory=256Mi \
  --cpu=1 \
  --concurrency=80

# Get your live URL
gcloud run services describe my-app \
  --region=us-central1 \
  --format="value(status.url)"
✓ Why this step? --min-instances 0 is the key free-tier flag — your service hibernates when idle and only wakes on request. Cloud Run handles HTTPS, load balancing, and scaling automatically.

YOUR NOTES

Even on the free tier, setting a $0 budget alert is good practice. It notifies you immediately if something unexpected generates charges. You can also set a hard cap using Cloud Billing budget actions to stop billing if a threshold is crossed. 

# Enable the billing budgets API
gcloud services enable billingbudgets.googleapis.com

# Create a $5 monthly budget with alerts at 50% and 100%
# (Replace BILLING_ACCOUNT_ID — get it from: gcloud billing accounts list)
gcloud billing budgets create \
  --billing-account=BILLING_ACCOUNT_ID \
  --display-name="Portfolio Budget" \
  --budget-amount=5USD \
  --threshold-rule=percent=50,basis=CURRENT_SPEND \
  --threshold-rule=percent=100,basis=CURRENT_SPEND
✓ Why this step? Free tier limits are generous but a misconfigured loop or traffic spike could push you over. A $5 budget alert costs nothing to set up and means no surprise bills.

YOUR NOTES

Once deployed, grab the auto-generated Cloud Run URL and paste it into your portfolio card's "View Project" link. For a cleaner URL, you can also map a custom domain to your Cloud Run service — free, takes effect within minutes. 

# Fetch your live service URL
gcloud run services describe my-app \
  --region=us-central1 \
  --format="value(status.url)"
# → https://my-app-abc123-uc.a.run.app

# (Optional) Map a custom domain
gcloud run domain-mappings create \
  --service=my-app \
  --domain=app.yourdomain.com \
  --region=us-central1
<!-- Update your portfolio card href with the live URL -->
<a href="https://my-app-abc123-uc.a.run.app" class="card-link">
  View Project →
</a>
✓ Why this step? Cloud Run provides a free HTTPS subdomain automatically. Custom domains are also free and need only a DNS record update. Your portfolio card now links directly to your live project.

YOUR NOTES

For a static HTML file like this portfolio, Cloud Storage is simpler and cheaper than Cloud Run — no Docker, no containers, no build step. The free tier gives you 5GB storage and 1GB egress/month, which is more than enough. Once set up, deploying an update is a single command. 

# 1. Create your bucket in EU region (name is already reserved)
gsutil mb -l eu gs://daniel-adeniran-portfolio

# 2. Make all files in the bucket publicly readable
gsutil iam ch allUsers:objectViewer gs://daniel-adeniran-portfolio

# 3. Tell GCP this bucket serves a static website (index.html is the entry point)
gsutil web set -m index.html gs://daniel-adeniran-portfolio

# 4. Upload your index.html for the first time
gsutil cp index.html gs://daniel-adeniran-portfolio

Your site will be live at:

# Your live portfolio URL
https://storage.googleapis.com/daniel-adeniran-portfolio/index.html

Every time you receive an updated index.html from Claude, just run this one command to push it live — that's the only step needed for every future update: 

# Re-deploy after any update — run from the folder containing index.html
gsutil cp index.html gs://daniel-adeniran-portfolio

# Verify it uploaded correctly
gsutil ls -l gs://daniel-adeniran-portfolio/index.html
# Expected: file size, timestamp, and gs://daniel-adeniran-portfolio/index.html
✓ Why Cloud Storage instead of Cloud Run? Cloud Run is for apps that need a server to run code. This portfolio is a static HTML file — it doesn't need a server. Cloud Storage serves it directly, which means zero cold starts, no Dockerfile, no container registry, and a simpler one-command deploy workflow. The free tier comfortably covers a portfolio site with no ongoing cost.

YOUR NOTES

GitHub gives you two things here: version history — every update we make to your site is tracked as a commit, so you can always roll back — and GitHub Pages, a free hosting option that gives you a clean URL like https://daniel-adeniran.github.io/portfolio. That URL is better for LinkedIn and recruiters than the Cloud Storage URL. You'll maintain both.

Part 1 — Create the repository on GitHub.com
Do this in your browser before running any terminal commands.

ⓘ Before you start You need a GitHub account. If you don't have one, sign up free at github.com then come back here. Your GitHub username will appear in your public portfolio URL so choose it carefully. 
# Steps to create the repo on github.com (do this in your browser):
# 1. Log into github.com
# 2. Click the + icon (top right) → "New repository"
# 3. Repository name: portfolio
# 4. Set visibility to: Public (required for free GitHub Pages)
# 5. Do NOT tick "Add a README file"
# 6. Click "Create repository"
# 7. Copy the HTTPS URL shown — it will look like:
#    https://github.com/YOUR-USERNAME/portfolio.git

Part 2 — Connect Cloud Shell to GitHub
GitHub requires authentication before you can push. The quickest way from Cloud Shell is a Personal Access Token. Generate one in your browser first. 

# Generate a Personal Access Token on github.com:
# 1. Click your profile photo (top right) → Settings
# 2. Scroll to the bottom of the left sidebar → Developer settings
# 3. Personal access tokens → Tokens (classic) → Generate new token (classic)
# 4. Note: "Cloud Shell deploy"
# 5. Expiration: 90 days (or No expiration for a portfolio)
# 6. Tick the "repo" scope checkbox
# 7. Click Generate token
# 8. COPY THE TOKEN NOW — GitHub only shows it once
#    It looks like: ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

Part 3 — Push your site to GitHub from Cloud Shell
Run these commands in Cloud Shell. When Git asks for your password, paste your Personal Access Token — not your GitHub password. 

# Navigate to the folder containing your index.html
cd ~

# Create a new folder for your portfolio repo and move index.html into it
mkdir portfolio
cp index.html portfolio/
cd portfolio

# Initialise Git in this folder
git init
git branch -M main

# Set your Git identity (use the email registered on GitHub)
git config user.name "Daniel Adeniran"
git config user.email "your-email@example.com"

# Connect to your GitHub repo (replace YOUR-USERNAME with your GitHub username)
git remote add origin https://github.com/YOUR-USERNAME/portfolio.git

# Stage and commit your file
git add index.html
git commit -m "Initial portfolio site"

# Push to GitHub (enter your GitHub username, then paste your token as password)
git push -u origin main
# Verify the push worked — your file should appear at:
# https://github.com/YOUR-USERNAME/portfolio

# Check your remote is set correctly
git remote -v
# Expected:
# origin  https://github.com/YOUR-USERNAME/portfolio.git (fetch)
# origin  https://github.com/YOUR-USERNAME/portfolio.git (push)

Part 4 — Enable GitHub Pages
Do this in your browser once the push in Part 3 succeeds. 

# Enable GitHub Pages on github.com:
# 1. Go to your repo: github.com/YOUR-USERNAME/portfolio
# 2. Click the Settings tab (top of the repo)
# 3. Scroll down the left sidebar → Pages
# 4. Under "Branch" — select: main
# 5. Folder: / (root)
# 6. Click Save
# 7. Wait ~60 seconds then visit your live site at:
#    https://YOUR-USERNAME.github.io/portfolio

Part 5 — Updating the site after future changes
Every time you download an updated index.html from Claude, run both commands below — one pushes to GCP, one pushes to GitHub. Both URLs stay in sync. 

# Copy the new file into your portfolio folder
cp ~/index.html ~/portfolio/index.html
cd ~/portfolio

# Push to GitHub (updates GitHub Pages automatically)
git add index.html
git commit -m "Update portfolio site"
git push

# Push to GCP Cloud Storage (updates your GCP URL)
gsutil cp index.html gs://daniel-adeniran-portfolio
✓ Why both GitHub and GCP? GitHub Pages gives you a clean shareable URL for LinkedIn and CVs. GCP Cloud Storage keeps your site on infrastructure you control and already know. Git history means every version of your portfolio is recoverable — useful as the site grows over time with new projects, certs, and SQL or Python work.
